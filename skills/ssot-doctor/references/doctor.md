# Doctor Verification

Doctor verification checks whether existing SSOT content is still trustworthy. It does not catch up new commits/sessions; catching up new changes is the job of [`commit-audit.md`](../../ssot-audit/references/commit-audit.md) and [`conversation-audit.md`](../../ssot-audit/references/conversation-audit.md) respectively.

Use cases:

- Default companion step of proactive catch-up.
- CI pipeline or manual `ssot doctor` execution.
- About to declare Doctor `passed` / `no-op` / "no update needed".
- Need to verify whether `covered`, `converged` or tracked-waterline claims have evidence support.

Doctor's `passed`, `no-op` or "no update needed" conclusions are stop conclusions and must go through an independent reviewer.

## 1. Execution method

```text
1. Read SSOT/STATUS.md:
   - tracked_commit
   - tracked_session
   - tracked_skill_version
   - documentation_language
   - documentation_language_evidence
   - coverage states
   - open adjudications / open gaps / stop-review gate

2. Scan existing SSOT content per this file's checklist.
   - Finish L1 deterministic checks first.
   - Enter L2 semantic checks after L1 fully passes.

3. Output the stale list and recommended fix actions.

4. The agent may fix directly, or report to the user for decision.

5. If about to declare Doctor passed / no-op / "no update needed", request an independent reviewer to audit Doctor scope.
   - no-more-required-changes: may record Doctor pass in the STATUS.md stop-review gate.
   - needs-fix: apply remaining changes and re-run Doctor and reviewer.
```

In standalone scenarios Doctor runs only verification checks; it does not run commit/session audits. In proactive catch-up Doctor usually runs after commit/session audits to verify the trustworthiness of existing SSOT against current state.

## 2. Checklist

The checklist has two layers. L1 deterministic checks can be executed quickly by the agent (results are pass/fail) and do not need an independent reviewer for each item. L2 semantic checks need agent judgement and an independent reviewer. Doctor executes L1 first, then enters L2 after L1 fully passes.

### 2.1 L1 deterministic checks

L1 check criteria are structural: whether a field exists, whether a path is valid, whether a value is legal. Results are deterministic pass/fail.

## 3. Coverage hard blockers

When Doctor hits any of the following, the related product, architecture, testing, or other affected area cannot be marked `covered`. This blocks only SSOT coverage state, not daily code development.

- `architecture/README.md` is missing, or that file cannot build a quick mental model nor route to authoritative views/domains.

```text
[TEMP-SURFACE] tech-debt/0002-compat-parser.md -- temporary_surface=true but owner/reason/verification_guard missing
[COVERED-PLACEHOLDER] product/README.md -- area is covered but owner still contains starter placeholder text

[GAP-OWNER] STATUS.md -- open gap row says "create tech debt later" instead of linking DEBT-NNNN or another owner
[CAPTURE-LIFECYCLE] STATUS.md -- resolved/passed capture section still contains "Pending action"; promote, defer with owner/trigger, or expire
[SILENT-DEFERRAL] tech-debt/0003-example.md -- future-work wording without owner/reference or retrigger signal
[MARKDOWN-FENCE] architecture/README.md -- fenced code block opened but not closed
```

Tag semantics:

- `[STALE]`: reference, path, fact or state is stale.
- `[CONFIDENCE]`: confidence tag illegal value, inconsistent with write location, or hypothesis/candidate blocking covered.
- `[MISMATCH]`: SSOT inconsistent with code/config/schema/test current facts.
- `[INDEX]`: index state inconsistent with entry state.
- `[ADJUDICATION]`: needs adjudication but queue missing or state wrong.
- `[DIAGRAM]`: architecture diagram protocol not satisfied, including required Mermaid diagrams missing, insufficient state/evidence, mixed current/target, or external diagrams used directly as authoritative.
- `[DECOMPOSITION]`: architecture decomposition has no evidence-guided signals, only mirrors directories/package names/class names/teams or external topic trees, lacks rejected axes / false friends, or a domain lacks effective `why separate`.
- `[SYNTHESIS]`: root/view writes sweeping conclusion without domain evidence, view takes over domain detail, or unverified content not marked `gap` / `unknown`.
- `[OWNER-ANCHOR]`: same long-lived fact maintained in multiple places, key claim lacks a unique authoritative owner, or Reader Map / root summary holds independent facts.
- `[PRODUCT]`: required product skeleton missing, or PRD, product promise, capability, journey, roadmap, or product acceptance not absorbed into the product owner.
- `[PRODUCT-OWNER]`: product Reader Map / owner index copies body facts, capability/journey over-split, or product facts lack a top stable owner.
- `[PRODUCT-ARCH-DRIFT]`: architecture rewrites, copies or changes facts owned by the product owner, or product files own runtime/API/SDK/schema implementation detail; keep product as intent layer, architecture as implementation response.
- `[ARCH-ROOT]`: architecture root entry cannot build a quick mental model and Runtime Owner Map, or all detail piled at root.
- `[ARCH-VIEWS]`: cross-owner technical views are missing, source material still holds technical operating model / critical runtime flow / CTG, or a view takes over domain-owned detail.
- `[ARCH-DOMAINS]`: state/resource/contract/failure/verification detail not in the domain authoritative location, domain lacks validity evidence, or domain shape is checklist-heavy instead of runtime-owner based.
- `[DESIGN-INTENT]`: root/view/domain lacks design intent, priorities, non-goals, success criteria, constraints or trade-off narrative, or consists mainly of tables.
- `[READER-MAP]`: entry map missing, stale, points at non-authoritative location, or holds independent long-lived facts in the entry.
- `[READABILITY]`: a user-facing SSOT body file violates prose-before-tables, positive-definition-on-first-use, or cell-is-not-a-paragraph (3+ cross-references / nested clauses / inline `BUG-`/`DEC-`/`ADJ-` reference plus a full verb clause), or key claims lack one-sentence conclusion, owner, why/risk/constraint or evidence. See `ssot-bootstrap` §3.7.
- `[KISS]`: an SSOT owner is technically structured but not simple to read: tables carry the main explanation, the same fact is repeated in sibling sections, or a register carries paragraph reasoning/checklists instead of pointers.
- `[KISS-TABLE-DENSITY]`: deterministic heuristic for table-line density in a user-facing SSOT body file. It is a triage signal, not proof; Doctor must inspect whether the table density is justified by index/lookup use.
- `[TESTING-LEDGER]`: `testing/` contains chronological validation history, command transcripts, date-by-date pass/fail logs, or "latest verification" rows whose only purpose is proving a batch ran. `testing/` may keep stable strategy, selection matrix, gates, fixtures, current baseline, known gaps, and defensive-test map entries with evidence pointers.
- `[STATUS-NOTES-LEDGER]`: `STATUS.md` Notes carries derived child-entry state, child counts, latest validation, dated batch history, current green/red state, command pass/fail history, or proof-of-work chronology rather than coverage pointers.
- `[INDEX-DERIVED-STATE]`: README/index/Reader Map files mirror entry-owned facts, generated counts, recent activity, verification rows, or lifecycle state beyond protocol-authorized minimal index fields.
- `[SHADOW-LEDGER]`: a non-owner SSOT file maintains chronological proof-of-work, task/run/status logs, latest-run rows, or dated incident history that belongs in an authorized owner such as stop-review evidence, bug entries, release notes, CI artifacts, commits, or entry files.
- `[ACTIONABILITY]`: a user-facing owner or high-risk entry does not help a future agent decide when to read it, where to inspect first, what not to do, or how to verify closure.
- `[ENTRY-ACTIONABILITY]`: a numbered `bugs/` or `tech-debt/` entry lacks a compact quick-entry surface for symptom/trigger, first checks, do-not-do, minimal verification, or current status/evidence.
- `[STATUS-STOP-REVIEW-LEDGER]`: `STATUS.md` stop-review fields carry full batch transcripts, command ledgers, or proof-of-work prose instead of concise conclusions and evidence pointers.
- `[READABILITY-LONG-LINE]`: a deterministic readability heuristic found a very long Markdown line; review whether paragraph content is compressed into a table cell or note.
- `[EVOLUTION]`: architecture evolution/migration/do-not-revive concept missing.
- `[BUG-GRANULARITY]`: high-impact bugs recorded too coarsely.
- `[SOURCE]`: long-lived facts in source material not absorbed.
- `[SOURCE-INVENTORY]`: root public Markdown or `docs/**/*.md` lacks a lifecycle header, STATUS inventory row, or audited exclusion.
- `[SOURCE-LIFECYCLE]`: working/historical/external source material carries strong current-fact language without authority downgrade fields such as `absorbed_to` and `do_not_use_for`.
- `[SOURCE-EXCLUSION]`: an audited exclusion row is missing `pattern`, `reason`, `owner`, `last_checked`, or `review_trigger`.
- `[THIN-DOCS]`: README/docs hold independent long-lived facts, or a public thin doc lacks a named SSOT owner.
- `[ARCH-CHECKLIST-HEAVY]`: architecture root/domain/view looks driven by a universal section checklist instead of Runtime Owner Map; lint is heuristic and Doctor should confirm.
- `[ADAPTER]`: SSOT-generated thin-adapter file-form issue: content exceeds size discipline, source hash drift, or adapter summary out of bounds. Handwritten / mixed startup files are not reported under this tag for lacking a generated marker.
- `[CORE-REF]`: core-reference-document fact issue: commands, directory map, workflows, architectural constraints, model/config rules or test strategy in startup/rules files not aligned with code/SSOT/protocol, or missing `STATUS.md` core-reference-document review record.
- `[SKILL-PROTOCOL]`: protocol waterline missing or behind.
- `[DOC-LANGUAGE]`: language lock missing, evidence insufficient, text deviates or language switch unadjudicated.
- `[CONSUMPTION]`: SSOT consumption chain broken -- startup reference file does not point to `SSOT/` or `$ssot-*`, SSOT path unreachable, or `SSOT/README.md` navigation entry missing.
- `[PLAYBOOK]`: an architecture domain owns mechanical task branches (≥3 ordered operations) but lacks a `playbook.md` modeled on `sdk-agent-runtime/playbook.md` (§0 startup / §1–N task branches / §3 gates / §4 debug ladder / §5 DoD / §6 prohibitions).
- `[SYMBOL-PIN]`: an invariant / contract row in an architecture README has no `path:src/...:LNN` or `tests/...::test_*` anchor a cold agent can resolve from one hop.
- `[SURFACE-PIN]`: a contract row lacks the user-observable surface anchor — API route + handler `path:LNN`, SQL identifier + table/column, DOM selector + component + Playwright test, or CLI command location. Product rows that are explicitly `state: design`, `state: debt`, `Out`, or `not_applicable` are not contract rows yet, but they must carry a current-truth row with current behavior and closure owner; missing state/closure routes to `[INTENT-RECOVERY]` / `truth-state-gap`.
- `[FAILURE-TRACE]`: a failure / recovery row names neither a regression test nor a `BUG-NNNN` entry, so the failure cannot be traced to evidence.
- `[STATE-TAG]`: an invariant / capability / contract lacks the inline `state: contract|design|poc|debt` lifecycle tag.
- `[FORK]`: the same long-lived fact (overlapping text, shared `path:LNN`, or same contract surface) is maintained in ≥2 architecture domain READMEs. (Doctor row `15D`, renumbered from 14W in v2.43.)

## 5. Pass / No-op review

If Doctor finds no issue, it cannot directly write `passed`. An independent reviewer must re-check Doctor's check scope.

The reviewer should read:

- `STATUS.md` and the stop-review gate.
- Doctor output.
- Sampled SSOT areas, especially `covered` areas.
- Related code/config/schema/test, source material, or diff/transcript evidence.
- Protocol upgrade notes corresponding to `tracked_skill_version`.

The reviewer returns:

- `no-more-required-changes`: may record Doctor pass or no-op in the `STATUS.md` stop-review gate.
- `needs-fix`: must list remaining changes; after fixing, re-run Doctor and reviewer.

Without an independent review, Doctor results can only serve as check output and cannot support `passed`, `no-op`, `covered`, `converged` or tracked-waterline advance.

## 6. Boundary with other flows

- **Commit audit**: catches up `tracked_commit..HEAD` code changes; see [`commit-audit.md`](../../ssot-audit/references/commit-audit.md).
- **Session audit**: catches up old transcripts after `tracked_session`; see [`conversation-audit.md`](../../ssot-audit/references/conversation-audit.md).
- **Protocol upgrade review**: catches `tracked_skill_version` up to the current `ssot-preflight` `metadata.protocol_version`; see [`protocol-upgrades.md`](../../ssot-audit/references/protocol-upgrades.md).
- **Knowledge integrity protocol**: full rules of the confidence state machine; see [`knowledge-integrity.md`](../../ssot-preflight/references/knowledge-integrity.md).
- **Thin-adapter protocol**: adapter content spec and generation rules; see [`adapter-strategy.md`](adapter-strategy.md).
- **Consumption audit**: evaluates the trigger/usage effectiveness of SSOT in real conversations (L4 behavioural probe), reverse-optimizing the trigger side; see [`consumption-audit.md`](consumption-audit.md). Doctor verifies content trustworthiness, consumption audit verifies trigger effectiveness; the two are orthogonal.
- **Doctor**: verifies trustworthiness of existing SSOT content; does not proactively interpret new diffs or new transcripts.

Proactive catch-up may combine these flows in any order, but every stop conclusion and tracked-waterline advance must have an explicitly-scoped independent stop review.
